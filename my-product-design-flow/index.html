我的产品设计流程与套路

发表于 2017-07-23 | 阅读次数 | 浏览 次

最近手头项目比较多，而且档期都很赶，千头万绪，手忙脚乱。虽然说需求的优先级都很高，但是再乱也不能乱节奏啊。

必须在一片混乱当中调整节奏，理顺思路，因此整理了一下我常用的产品设计流程与套路。

确立目标

在发现一个产品机会的时候，我首先会从客观上回答下面的问题：

  • 这个产品的目标用户是谁？
  • 最主要的场景是什么？
  • 这个场景之下，要解决什么问题？
  • 怎么解决？
  • 确定我的思路能解决这个问题之后，重新审视这个场景的强度、频次是否足够高？

上述的客观问题可被证实或证伪后，再回答一些主观上的问题：

  • 你和你的团队是否有能力及时地做出一个产品来解决这个问题？
  • 产品面世后，是否有后续运营手段来帮助产品触达用户，提高活跃？
  • 你的资金是否能足够支撑这些运营手段以达到目标效果？

不同类型的产品面临的问题可能会有所不同，有些特殊产品可能需要考虑更多的情况，例如，若涉及到政府公共服务的需求，就要考虑如何谈判，以及多方业务的对接和配合。

当回答完上述所有问题后，代表你已经有着较为明确的产品明标和方向，你的脑海里也开始浮现出产品的雏型。

调研

明确了目标后，就需要查找该产品类型的相关竞品，然后大量地试用和体验。

  • 在体验竞品的过程中，找出让你印象深刻的功能点，并看看是否该产品的主打功能；
  • 对比多个竞品之间相似的主打功能，结合场景，深刻体验和总结这些功能的实际效果；
  • 对比分析竞品的内容浏览体验，若有用户产出的内容（例如 PGC、UGC 的 feed 流，评论等），重点关注其质量和风格；
  • 如果该主打功能对你设定产品的方向和细节有帮助，并且来自顶尖竞品，有足够数据量支撑，那就需要手动统计或自动抓取其内容更新量、浏览量、互动数，并根据需要作出分析（担心其数据作假是没用的，除非你能获取竞品内部数据）；
  • 搜索顶尖竞品的相关报道和背景资料，观察各分发渠道的下载排名曲线变化，甚至翻一下官网，都能摸到不少门道。

对比分析完几十个同类竞品之后，如果你发现没一个竞品能很好地解决问题，而你的思路却能解决问题（或者说，有信心能解决问题），那就可以开始原型设计了。

设计

工具有很多，哪个顺手用哪个，本人最常用的是 Sketch 和 Axure，两者互补。

最近做了一些自己的小项目，有用到墨刀，演示和分享非常方便，但是 bug 也不少（2017.7.23 之前的版本），建议暂时只用来做演示和分享，不建议用来画原型和存档。具体方法就是先用 Axure 或 Sketch 来画页面原型，然后导出为图片后上传至墨刀，再通过热点来做页面跳转。

具体怎么画原型，我就不多说了，教程一搜一吨，且熟能生巧。这里给一些经验和建议：

  • 将原型和产品说明文档合并来做，左边是页面原型，右边是该页面的交互效果、算法描述、数据接口、异常情况等说明，既方便 UI、研发、测试同学的查看，又方便迭代；
  • 不做动画效果和颜色处理，有好的想法可以先用简单清晰的文字来描述，然后跟专业的 UI 同学讨论决定；
  • 规范编写页面的标题，在页面原型之外，再补充一份页面流程，并清晰地标注好页面间的逻辑判断和跳转关系。

在我的经验中，产品文档永不可能一步到位，力求快速迭代。对于原型和文档的初稿来说，不需要太抠细节，只需要流程清晰，逻辑没有硬伤，就可以快速进入评审阶段。

评审

准备好原型和文档之后，就可以组织团队成员一起讨论，这就是评审。

在讨论的过程中，让团队成员提出疑问，能解决的问题当场解决，不能解决的在评审后再详细讨论和修改，不能打乱评审会节奏，必须在评审会上完成目标功能的场景演示或体验。

若在修改设计中推翻了某些需求，或作出了重大修改，需要将变更原因和细节通过邮件同步给全体成员，或再次组织评审会议。

交付

在最后一次评审之后，再规划版本节奏，个人感觉这样比较靠谱。因为经过反复修改后，基本已经知道了功能点的实现成本，哪些功能可以做，哪些需要延后，心中了了。

规划好版本之后，合理安排时间节奏，严格按照 deadline 输出最终产品说明文档，然后交付给 UI 设计和研发流程。

总结

以上就是我的产品设计流程，还有一些小套路，一直运用在日常工作当中。在小型项目中，某些步骤可能会简单一些，例如直接通过邮件来完成评审，但是其核心流程是不变的。

走完一个流程之后，就可以开始进入下一轮的迭代了。
